## Implementation order

The dependencies flow as:
1. **Sidecar first**: Update `/send-message` response shape before updating the C# client (avoids mismatch window).
2. **Backend exceptions**: Add new exception types before modifying client/processor/dispatcher.
3. **Backend client**: Update `WhatsAppServiceClient.SendAsync()` to throw typed exceptions.
4. **Backend processor/dispatcher**: Update catch blocks and timeout rethrow.
5. **Health cache**: Add `WhatsAppHealthCache`, update monitor and status endpoint.
6. **Idempotency key**: Update `CreateChild`, update tests.
7. **Templates/helper**: Add `PayloadVersion`, refactor template lookup.
8. **Frontend**: Add i18n keys first, then confirmation dialog, then type updates.
9. **Validation**: `dotnet test` passes; `npm run build` passes; `npm run i18n:check` passes.

- Timeout → message status = Pending (not Failed)
- Circuit-open → Pending, `Attempts` not incremented

## Key invariants to preserve

- `parentMessageId` field MUST remain in `WhatsAppAttemptDto` (FR-024) — do not remove it.
- No new EF Core migration (FR-027) — `PayloadVersion` goes in JSON payload, not a DB column.
- No new npm or NuGet packages.
- The existing exponential backoff formula (`Min(4^attempts * 30s, MaxBackoff)`) and `MaxAttempts` configuration are unchanged.
- `npm run i18n:check` MUST pass — add keys to both locales simultaneously.
